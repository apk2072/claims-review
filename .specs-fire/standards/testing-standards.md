# Testing Standards

## Overview

Testing goals for this project are pragmatic, not exhaustive: prove each pipeline stage and API route behaves correctly against mocked AWS services, without ever touching real PHI or requiring live AWS credentials for the test suite to pass.

## Testing Framework

**Framework**: pytest (Python — pipeline, backend, agent, infra); Vitest + React Testing Library (frontend)
**Runner**: `pytest` / `npm run test`

## Test Types

| Type | Tool | Location | When to Use |
|------|------|----------|-------------|
| Unit | pytest | `{component}/tests/unit/` | Pure logic: confidence blending, verdict routing, prompt formatting |
| Integration | pytest + moto | `{component}/tests/integration/` | Lambda handlers, FastAPI routes, Aurora access — with AWS mocked via moto |
| Frontend component | Vitest + RTL | `frontend/src/**/*.test.tsx` | Reviewer UI components, verdict actions |
| Manual/E2E | preview tools + real AWS dev stack | n/a | Full pipeline run on synthetic sample claims before considering a work item done |

## Coverage Requirements

**Target**: 70%
**Enforcement**: informal — checked at the end of each work item, not gated in CI (no CI configured yet)

**Critical paths that MUST have coverage:**
- Composite confidence score calculation
- Auto-verdict threshold routing (both sides of the 0.92 boundary)
- Reviewer verdict atomic write (status update + audit row + metric + trace tag)
- Prompt promotion gate logic (safety floor, correctness floor, bootstrap CI)

## Test Naming

**Pattern**: `test_{unit_under_test}_{scenario}_{expected_outcome}`

**Examples**:
- `test_composite_score_low_completeness_reduces_score` — confidence blending honors completeness weight
- `test_verdict_routing_above_threshold_auto_approves` — gold-layer routing boundary condition
- `test_promotion_gate_safety_regression_blocks_promotion` — hard safety floor enforced

## Test Structure

```python
def test_verdict_routing_above_threshold_auto_approves():
    # Arrange
    doc = make_claim_document(composite_confidence=0.95)

    # Act
    verdict = route_verdict(doc, threshold=0.92)

    # Assert
    assert verdict.is_automated is True
```

## Mock Strategy

**Approach**: `moto` for all AWS service calls (S3, Textract, Bedrock via stubbed boto3 client, Step Functions) in unit/integration tests. Bedrock/Textract responses are captured once as fixture JSON from a real dev-account call, then replayed in tests.

**Guidelines**:
- Never call real Bedrock/Textract endpoints in the automated test suite (cost + nondeterminism)
- Fixture files live in `{component}/tests/fixtures/`, named after the scenario they represent
- Real AWS calls only happen in manual verification runs against the dev stack, with synthetic documents

## Test Data

**Strategy**: synthetic, generated claim documents only — never real PHI, even fake-looking real patient data. Use clearly fake names (e.g., "Jane Test Doe") and fabricated identifiers.

**Guidelines**:
- Keep a small fixed set of sample synthetic claim PDFs/images in `pipeline/tests/fixtures/sample_claims/` covering: clean high-confidence doc, blurry/low-confidence doc, missing-field doc, wrong-document-type doc
- Do not generate new synthetic PHI-shaped data with real-sounding SSNs/MRNs — use obviously fake patterns

## Running Tests

```bash
# Run all tests
pytest

# Run with coverage
pytest --cov=. --cov-report=term-missing

# Run specific test file
pytest pipeline/tests/unit/test_confidence_scoring.py

# Run in watch mode (frontend)
npm run test -- --watch
```

## CI/CD Integration

CI/CD integration not configured. Tests are run manually before marking a work item complete.
