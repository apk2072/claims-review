## Key Implementation Details

### 1. Two different DB connection paths by design

`common/db.py` (used by VPC-internal services) expects a normal `psycopg`-flavored `DATABASE_URL`. `common/migrations/env.py` (used by Alembic, run from this machine) builds a `postgresql+auroradataapi://` URL instead, reading `AURORA_CLUSTER_ARN`/`AURORA_SECRET_ARN`/`AURORA_DATABASE_NAME` from the environment. This isn't incidental complexity — it's the direct consequence of work item 2's deliberate choice to give Aurora no public endpoint.

### 2. Hand-written migration, not autogenerated

`alembic revision --autogenerate` doesn't work over the Data API — its schema-reflection query trips `UnsupportedResultException` because Data API can't deserialize the `CHAR` result type that `information_schema.columns` reflection produces. Irrelevant for a from-scratch schema (nothing to diff against), so the initial migration was written by hand as a sequence of `op.create_table()` calls. Documented as a known limitation in `common/README.md` for whoever writes the *second* migration.

### 3. pgvector needed an explicit bind-time cast

The Data API sends bind parameters without Postgres's normal type-OID negotiation (which a real `psycopg` connection does automatically over the wire protocol). An un-cast `vector` column insert therefore arrives as plain text and Postgres rejects it outright — a hard failure caught by the integration test, not a silent-corruption risk. Fixed with a small `Vector` subclass in `common/models.py` that adds `bind_expression` → `CAST(value AS vector(...))`. This is a no-op for real `psycopg` connections, so it's used as *the* vector type everywhere, not conditionally.

### 4. Verdict/status columns are plain strings, not Postgres ENUMs

`claims.status` and `reviews.verdict` are `String` columns with allowed-value tuples documented in Python (`CLAIM_STATUSES`, `REVIEW_VERDICTS`), not native Postgres `ENUM` types — adding a new status later is a one-line Python change instead of an `ALTER TYPE` migration.

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Alembic connection method | RDS Data API | No public endpoint on Aurora; consistent with work item 2's pgvector verification approach |
| Initial migration authoring | Hand-written | Autogenerate's reflection query is incompatible with the Data API |
| pgvector bind casting | Custom `Vector` subclass with explicit cast | Data API skips type-OID negotiation; the cast is safe for `psycopg` too |
| Status/verdict columns | Plain `String`, not `ENUM` | Avoids `ALTER TYPE` migrations for new values |

## Developer Notes

- **`alembic revision --autogenerate` will not work against this Aurora cluster** — always write migrations by hand, or autogenerate against a throwaway local Postgres and adapt the output.
- **Always use `common.models.Vector`, never `pgvector.sqlalchemy.Vector` directly**, for any pgvector column in this codebase — the bare pgvector type will silently work over `psycopg` but fail over the Data API, which is an easy trap for whoever adds the next vector column.
- Expect a `DatabaseResumingException` (or its Data-API-driver-wrapped equivalent) on the first Aurora call after any idle period — retry once after ~15s. This showed up repeatedly across both this work item and work item 2; it's the normal scale-to-zero cold start, not a bug.
- The AWS MCP server (`aws-mcp`, via `mcp-proxy-for-aws`) reconnected mid-session but its tools weren't visible to this conversation without a fresh session — if a future session has it available, prefer its `read_documentation`/`search_documentation` tools over guessing from package internals (as was done here to figure out the `aurora-data-api` URL/dialect behavior).
