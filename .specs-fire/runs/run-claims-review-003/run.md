## Decisions
- **Alembic connection method**: RDS Data API (aurora-data-api + sqlalchemy-aurora-data-api dialect) instead of direct psycopg (Aurora has no public endpoint by design; Data API needs no VPC network path, consistent with how pgvector was verified in work item 2)
- **Initial migration authoring**: hand-written, not autogenerated (Data API cannot serialize the CHAR result type Alembic autogenerate reflection produces (UnsupportedResultException); not a problem for a from-scratch schema anyway)
- **pgvector bind casting**: custom Vector subclass with explicit ::vector cast via bind_expression (Data API skips Postgres type-OID negotiation so untyped vector parameters are rejected; the cast is a safe no-op for real psycopg connections too)
- **claims.status / reviews.verdict typing**: plain String columns, not Postgres ENUM (avoids ALTER TYPE migrations when new statuses/verdicts are added later)
